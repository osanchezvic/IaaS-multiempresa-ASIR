# Plan de Mejora de la Infraestructura SaaS
 
Este plan describe las acciones necesarias para mejorar la robustez, seguridad y mantenibilidad de la plataforma SaaS.

## 1. Análisis de Estado Actual
El proyecto actual es un conjunto de scripts bash que orquestan contenedores Docker. La estructura es funcional pero presenta riesgos operativos y de seguridad.

### Hallazgos Principales:
- **Riesgo de Datos:** `scripts/deploy.sh` realiza un `rm -rf` directo del directorio de servicio, lo que puede causar pérdida de datos si se ejecuta sobre una instalación existente.
- **Fragilidad en Templates:** El procesamiento de templates mediante `sed` es sensible a caracteres especiales y espacios.
- **Gestión de Credenciales:** Las contraseñas y secretos se almacenan en archivos JSON sin cifrado aparente en el almacenamiento local.
- **Concurrencia:** No parece haber mecanismos de bloqueo (locking) para evitar despliegues concurrentes que puedan corromper el estado.

## 2. Objetivos del Plan
- [ ] **Proteger datos:** Implementar un mecanismo de respaldo automático antes de operaciones destructivas.
- [ ] **Mejorar robustez:** Añadir manejo de errores más granular y validaciones de estado.
- [ ] **Seguridad:** Evaluar y fortalecer la gestión de secretos.
- [ ] **Mantenibilidad:** Refactorizar la lógica de templates y configuración.

## 3. Pasos de Ejecución

### Fase 1: Prevención de Pérdida de Datos
- Modificar `scripts/deploy.sh` para crear un respaldo comprimido (tar.gz) del directorio del servicio antes de realizar modificaciones destructivas.
- Implementar una función `rollback` que pueda restaurar el estado anterior en caso de fallo durante el despliegue.

### Fase 2: Robustez y Estabilidad
- Refactorizar `procesar_template` para evitar el uso de `sed` directo con variables que pueden contener caracteres especiales (usar `envsubst` o técnicas de escape más seguras).
- Implementar bloqueo de archivos (flock) en `deploy.sh` y `destroy.sh` para prevenir ejecuciones simultáneas y evitar colisiones de estado.

### Fase 3: Seguridad
- Asegurar que la función `guardar_credenciales` aplique permisos `600` (lectura/escritura solo para el dueño) a todos los archivos generados que contengan contraseñas, manteniendo el almacenamiento en texto plano según lo solicitado.

### Fase 4: Limpieza y Documentación
- Estandarizar las variables de entorno para que no dependan de rutas hardcodeadas (`/srv/`).
- Añadir logs detallados en caso de fallo de red Docker.
